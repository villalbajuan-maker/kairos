# V1 Pilot-Ready Spec

## Purpose

This document defines the first serious version of Kairos that should be placed in front of real users for controlled validation.

It is not the long-term product vision.

It is the minimum serious version that is worthy of real-world testing.

## Core principle

V1 is not a prototype in the weak sense.

It is a real, focused product slice designed to validate the central Kairos loop with real people.

That means V1 must be:

- narrow
- credible
- usable
- spiritually serious
- measurable
- bilingual in foundation

## Primary V1 question

Can Kairos guide a real believer from ambiguity to a clear decision and a concrete next step in a way that feels trustworthy, useful, and repeatable?

Everything in V1 should serve that question.

## V1 success definition

V1 is successful if real users can:

1. enter with a real decision
2. understand what Kairos is asking them to do
3. complete the guided journey
4. reach a concrete decision or explicit waiting state
5. leave with a defined next step
6. report that the process felt helpful, serious, and spiritually appropriate

## The minimum serious V1

V1 should include the full decision journey, but in a deliberately constrained form.

### What V1 must include

#### 1. One full Kairos session flow

The user must be able to move through:

- Phase 1 — Define the decision
- Phase 2 — Separate the internal noise
- Phase 3 — Seek God intentionally
- Phase 4 — Collect and reduce ambiguity
- Phase 5 — Decide and commit
- Phase 6 — Execute and sustain

#### 2. One decision at a time

V1 must enforce the discipline of one decision per session.

#### 3. Phase-aware interaction

The system must clearly behave differently by phase.

It must not feel like generic free chat.

#### 4. Structured ambiguity reduction

The product must be able to:

- listen to freeform input
- propose clearer formulations
- narrow the user into concrete language

#### 5. Guided spiritual engagement

V1 must include a serious and structured form of:

- prayer prompting
- Scripture reflection
- pause / return behavior

#### 6. Real closure

The session must end in one of these:

- a concrete decision and action
- an explicit decision to wait, with a defined next spiritual step

#### 7. Safety boundary representation

V1 must include basic escalation handling.

Unsafe cases must not proceed as normal discernment sessions.

#### 8. Commercial gate

V1 must include:

- free phase 1
- paywall after phase 1
- code-based access

This is not because monetization is the top priority.

It is because the product should validate the true shape of the business from the beginning.

#### 9. Basic measurement

V1 must allow the team to observe:

- phase completion
- paywall exposure
- unlock rate
- pause / return rate
- decision completion
- execution confirmation
- escalation frequency

#### 10. Bilingual support

V1 should support the same core guided process in:

- English
- Spanish

The bilingual requirement does not mean every expression must be perfect on day one.

It means the product should be intentionally built so real users can meaningfully complete the journey in either language.

## Recommended V1 use case scope

V1 should be broad enough to support real decisions, but narrow enough to remain disciplined.

Recommended initial approach:

- allow multiple decision categories in principle
- strongly bias early testing toward the primary use case:
  relationship continuation / closure

This preserves real-world usefulness without overpromising maturity across every type of case.

It should also allow pilot learning across both primary languages, even if one language receives the first heavier volume of testing.

## Recommended V1 product posture

V1 should feel like:

- a serious guided discernment session
- a structured conversation
- a product with gravity

V1 should not feel like:

- a chatbot demo
- a spiritual journaling toy
- a generic AI wrapper
- a content browser

## V1 user experience requirements

### Entry

The user must be able to arrive with a real situation and start immediately.

### Clarity path

The system must make visible progress from vague description to defined decision.

### Reflection path

The spiritual phase must feel intentional, not decorative.

### Commitment path

The user must be asked to commit to a specific next action.

### Return path

The user must be able to come back after pause and continue without losing the thread.

## What V1 should not include yet

V1 should not include:

- broad public growth mechanisms
- advanced recommendation systems
- deep personalization
- broad church admin tooling
- polished multi-user leader dashboards
- full Bible infrastructure
- many parallel interfaces
- complex billing logic

It should, however, include English and Spanish as core product languages.

These are valuable later, but they are not the first proof.

## V1 trust requirements

Before live pilot, the team should be confident that V1:

- represents Kairos faithfully
- does not trivialize the spiritual process
- does not overclaim spiritual authority
- handles ambiguity with enough discipline
- has minimum viable escalation behavior

## V1 pilot audience

The first users should come from:

- trusted pastors
- trusted leaders
- trusted faith communities
- people already facing real decisions

This should be a controlled release, not an open internet launch.

## V1 pilot distribution model

Recommended:

- code-based access
- leader-distributed entry
- small initial cohorts
- direct feedback loop

This keeps the learning environment high-trust and low-chaos.

## Pilot feedback questions

Every serious V1 test should help answer:

- Did the user understand what Kairos was doing?
- Did the process feel spiritually serious?
- Did the process help reduce ambiguity?
- Did the user reach a real decision?
- Did they trust the product enough to complete it?
- Would they use it again for another decision?
- Would a leader recommend it to someone else?

## V1 non-negotiables

Before calling V1 pilot-ready, these must be true:

1. The user can complete the full flow.
2. The product preserves phase integrity.
3. The product includes pause and return behavior.
4. The product includes basic escalation handling.
5. The product includes paywall and code access behavior.
6. The product captures enough analytics to learn from usage.
7. The product feels like Kairos, not like a generic assistant.
8. The core session can be completed meaningfully in English and Spanish.

## Decision rule for launch

If the team has to choose between:

- more polish
- or a more faithful core journey

choose the more faithful core journey.

V1 will succeed because the process works, not because the interface is over-finished.

## Final V1 statement

The first serious version of Kairos should be the smallest version that can honestly be placed in front of a real believer with a real decision and still represent the product with integrity.
